  <name>Task 6: `src/config.ts` — Zod-Schema, dotenv, exit(0) bei Fehlern</name>
  <files>src/config.ts</files>
  <action>
Lege `src/config.ts` an. Inhalt:

```typescript
import "dotenv/config";
import { z } from "zod";

/**
 * Zod-Schema für die 4 Pflicht-Env-Vars (siehe CFG-01).
 *
 * Alle als `min(1)`, weil leerer String genauso schlecht wie fehlend ist —
 * dotenv liefert leere Strings für definierte aber leere Keys (KEY=).
 */
const EnvSchema = z.object({
  LASTFM_API_KEY: z.string().min(1, "LASTFM_API_KEY fehlt oder ist leer"),
  LASTFM_USER: z.string().min(1, "LASTFM_USER fehlt oder ist leer"),
  GATHER_API_KEY: z.string().min(1, "GATHER_API_KEY fehlt oder ist leer"),
  GATHER_SPACE_ID: z.string().min(1, "GATHER_SPACE_ID fehlt oder ist leer"),
});

export type Config = z.infer<typeof EnvSchema>;

function loadConfig(): Config {
  const result = EnvSchema.safeParse(process.env);
  if (!result.success) {
    // Kein Logger-Import hier (Circular: logger braucht möglicherweise
    // selbst keine Config, aber defensiv halten wir den Boot-Pfad logger-frei).
    // Stderr direkt, klar formatiert.
    const issues = result.error.issues
      .map((i) => `  - ${i.path.join(".")}: ${i.message}`)
      .join("\n");
    process.stderr.write(
      `\n[config] Ungültige Konfiguration. Daemon wird NICHT neu starten.\n${issues}\n` +
        `\nLösung: Prüfe \`.env\` im Projekt-Root, vergleiche mit \`.env.example\`.\n\n`,
    );
    // EXIT CODE 0 (nicht 1!) — siehe CFG-03 und Pitfall 2:
    // launchd in Phase 4 nutzt KeepAlive: { SuccessfulExit: false, Crashed: true }.
    // Bei exit(1) würde launchd als "Crashed" werten und endlos restarten.
    // Bei exit(0) bleibt der Daemon gestoppt, bis User `.env` fixt und manuell startet.
    process.exit(0);
  }
  return result.data;
}

export const config: Config = loadConfig();
```

Wichtige Punkte (alle aus CONTEXT.md + Pitfall 2 + Pitfall 5):

1. **`import "dotenv/config"`** als erster Import — lädt `.env` automatisch in `process.env` per Side-Effect, BEVOR `EnvSchema.safeParse(process.env)` läuft.
2. **`process.exit(0)` bei Validation-Fehler** — NICHT 1. Begründung steht im Code-Kommentar, ist Phase-4-Constraint (DMN-02 KeepAlive-Strategie hängt davon ab).
3. **stderr-Ausgabe statt Logger** — der Logger (Task 7) wird NACH Config geladen, könnte aber eine Circular-Dependency provozieren, wenn er Config nutzt. Defensiv: Boot-Pfad logger-frei halten.
4. **Issues-Liste formatieren** — Zod liefert ein `issues`-Array, wir mappen das auf eine lesbare Liste mit Punkten. Klare Fehlermeldungen sind wichtig, weil der User die nur einmal sieht (Daemon stoppt danach).
5. **Top-Level `export const config = loadConfig()`** — Module-Load-Time-Validation. Importiert irgendwo `config`, wird beim ersten Import validiert. Klar deterministisch.
6. **Keine eigene `dotenv.config({ path })`-Variante** — `import "dotenv/config"` reicht für Phase 1 (Foreground-Run aus Repo-Root). Pitfall 20 (`launchd cwd != repo`) wird in Phase 4 mit `WorkingDirectory` in der Plist gelöst.

Type-Check: `npx tsc -p . --noEmit` muss durchlaufen, bevor der Task als done gilt.

Lege `src/logger.ts` an. Inhalt:

```typescript
import pino from "pino";

/**
 * Pino-Logger mit defensiver Redaction für API-Keys.
 *
 * Redaction-Strategie (CFG-04 + Pitfall 5):
 * - Konkrete Pfade (`env.GATHER_API_KEY`) für den Fall, dass jemand
 *   `log.info({ env: process.env }, "...")` schreibt.
 * - Wildcard-Pfade (`*.GATHER_API_KEY`) für jede beliebige Verschachtelung,
 *   z. B. `log.error({ details: { env: { GATHER_API_KEY: ... } } })`.
 * - `apiKey` und `api_key` als generische camel-/snake-case-Varianten —
 *   wenn z. B. ein Last.fm-Response-Body geloggt würde.
 * - `censor: "[REDACTED]"` macht den Eintrag im JSON sichtbar
 *   (statt komplett zu entfernen via `remove: true`) — beim Debugging
 *   ist das hilfreicher: man sieht, dass das Feld da war.
 *
 * Default-Level `info`, in Phase 4 ggf. via `LOG_LEVEL` env (QOL-02 v2).
 *
 * Destination: stdout (pino-Default). launchd routet beide Streams in
 * dieselbe Datei (siehe Phase 4 / Pitfall 11).
 */
export const log = pino({
  level: process.env.LOG_LEVEL ?? "info",
  redact: {
    paths: [
      "env.GATHER_API_KEY",
      "env.LASTFM_API_KEY",
      "*.GATHER_API_KEY",
      "*.LASTFM_API_KEY",
      "*.apiKey",
      "*.api_key",
    ],
    censor: "[REDACTED]",
  },
  timestamp: pino.stdTimeFunctions.isoTime,
});
```

Wichtige Punkte:

1. **Redact-Pfade mit Wildcards** — exakt das Pattern aus PITFALLS.md (Pitfall 5). `*.GATHER_API_KEY` matched jede Tiefe (`{a:{b:{GATHER_API_KEY:...}}}`).
2. **`censor: "[REDACTED]"` statt `remove: true`** — siehe Code-Kommentar. Sichtbarkeit beim Debugging > minimale Log-Größe.
3. **`level: process.env.LOG_LEVEL ?? "info"`** — kein zwingender Env-Var, default `info`. Erlaubt zukünftiges Override (`LOG_LEVEL=debug` für Dev).
4. **`pino.stdTimeFunctions.isoTime`** — ISO-8601-Timestamp statt Unix-Millis. Bessere Lesbarkeit in `tail -f ... | pino-pretty`.
5. **Keine Custom-Destination** — pino-Default ist stdout, launchd handlet das in Phase 4.
6. **KEINE `pino.transport({...})`-File-Rotation** — out of scope für Phase 1, kommt ggf. in v2 (siehe Pitfall 12, Empfehlung "Option c) für v1").
7. **Keine `pino-pretty`-Dependency** — der User kann beim Debugging `tail -f log | npx pino-pretty` nutzen, kein Build-Time-Pflicht.

Selbsttest (kein automated test, aber prüfbar): wenn man `log.info({ env: { GATHER_API_KEY: "abc123" } }, "test")` schreibt, sollte der Output `{"env":{"GATHER_API_KEY":"[REDACTED]"}}` enthalten — wird im Smoke-Test (Plan 02) implizit mitgetestet.

Type-Check: `npx tsc -p . --noEmit` muss durchlaufen.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `.env` → `process.env` → Code | Secrets verlassen die `.env`-Datei via dotenv. Risiko: `.env` versehentlich committen, `process.env` versehentlich loggen. |
| Code → Logs (`stdout`/`stderr`) | Logs könnten Secrets exfiltrieren. Backup-Tools (Time Machine, Cloud-Sync) ziehen Logs mit. |
| Code → Git-Repo | `.env` und `*.log` dürfen NIEMALS in Git landen. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | `.env` mit `GATHER_API_KEY` | mitigate | `.gitignore` als ALLERERSTER Commit (Task 1, vor jedem `git add`). Pre-Commit-Check: `git ls-files \| grep -E '\.env$'` muss leer sein. |
| T-01-02 | Information Disclosure | Pino-Logger könnte API-Keys loggen | mitigate | Redact-Paths mit Wildcards (`*.GATHER_API_KEY`, `*.apiKey`) plus konkrete (`env.GATHER_API_KEY`) — Task 7. Censor mit `[REDACTED]`. |
| T-01-03 | Denial of Service (self-DoS) | launchd-Crash-Loop bei Config-Fehler | mitigate | `process.exit(0)` (NICHT 1) bei Zod-Validation-Fehler — Task 6. KeepAlive-Strategie in Phase 4 wertet exit(0) als "nicht restarten". |
| T-01-04 | Tampering | `.env.example` mit echten Werten committet | mitigate | Task 2: `.env.example` enthält explizit nur `KEY=` ohne Werte. Verify-Check prüft das. |
| T-01-05 | Information Disclosure | Last.fm-API-Key in Logs | mitigate | Same redact-Strategie wie T-01-02 (`LASTFM_API_KEY` + Wildcards). Read-only-Risiko niedrig, aber konsistent behandelt. |
| T-01-06 | Spoofing | Gather-API-Key kompromittiert (read+write auf Space) | accept | Single-User-Tool, lokales `.env`. Keine User-Input-URL/Body, kein Server-Mode. Mitigation: Pre-Commit-Check + Redact + manuelle Key-Rotation bei Verdacht. |
